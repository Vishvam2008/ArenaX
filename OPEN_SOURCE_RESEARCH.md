# ArenaX Open-Source Research and Integration Report

Date: 2026-06-13

## Codebase Audit

ArenaX is currently a static single-page esports prototype with plain HTML, CSS, and JavaScript. The app state lives in memory, with admin accounts stored in `localStorage`. A separate zero-dependency Node HTTP service moves payment screenshots through `Payments/Pending`, `Payments/Approved`, and `Payments/Rejected`.

Existing strengths:

- Manual financial workflow is explicit: deposits, withdrawals, entry fees, rewards, and refunds require admin approval.
- QR deposit flow includes amount, QR display, UTR capture, screenshot proof, admin review, and audit trail.
- Tournament controls already cover status changes, room release, check-in, participant approval, refunds, disqualification, result verification, and reward queues.
- Admin tools include QR management, payment verification, tournament management, wallet controls, staff management, audit logs, and APK center.

Weak areas found:

- Production security is prototype-level: client-side admin auth, base64 password storage, in-memory data, broad CORS, and unsanitized HTML rendering.
- Admin dashboard has many useful controls but limited information architecture; it needs denser table views, stronger filters, saved views, and bulk operations.
- Tournament system is lobby/card based, not bracket-stage based. This is acceptable for Free Fire rooms, but not enough for future elimination formats.
- Payment server lacked file size limits, strict image validation, duplicate-safe writes, and environment-based configuration.
- No automated tests, no persistence layer, no backend authorization boundary, and no deployment/build pipeline.

## Evaluated Sources

### Drarig29/brackets-manager.js

Source: https://github.com/Drarig29/brackets-manager.js

Advantages:

- Mature tournament logic for round-robin, single elimination, double elimination, seeding, forfeits, match states, multiple stages, and storage adapters.
- Framework-neutral and storage-neutral, which fits a future ArenaX backend service.
- MIT license and active release history.

Disadvantages:

- Logic-only; no admin UI.
- Overkill for current Free Fire room/lobby workflows.
- Requires a real persistence layer before it becomes valuable.

Compatibility:

- Good future backend candidate once ArenaX moves from static prototype to API-backed tournaments.

Recommended components:

- Adopt later for bracket stages, seeding, match state transitions, final standings, and tournament storage contracts.

### Drarig29/brackets-viewer.js

Source: https://github.com/Drarig29/brackets-viewer.js

Advantages:

- Vanilla JS bracket display, themeable with CSS variables, compatible with any framework.
- Can render data generated by `brackets-manager.js`.

Disadvantages:

- Adds external UI surface not currently needed for Free Fire rooms.
- Would be awkward without bracket data generation and persistence.

Compatibility:

- Strong future fit for a bracket tab inside tournament details.

Recommended components:

- Use later for elimination bracket visualization, not for the current card/list tournament UX.

### Tabler

Source: https://github.com/tabler/tabler

Advantages:

- High-quality, responsive, open-source admin UI kit with cards, tables, forms, badges, and dashboard patterns.
- MIT license and large community.

Disadvantages:

- Bootstrap-based; directly importing it would conflict with the current custom CSS.
- Visual identity is general SaaS, not gaming-specific.

Compatibility:

- Excellent as pattern inspiration; poor as a direct dependency in the current static CSS architecture.

Recommended components:

- Dense admin metrics, table hierarchy, status badges, empty states, form grouping, and responsive admin layouts.

### CoreUI Free Bootstrap Admin Template

Source: https://github.com/coreui/coreui-free-bootstrap-admin-template

Advantages:

- Enterprise-grade admin shell patterns, forms, icons, and documented structure.
- Multiple framework options if ArenaX later migrates.

Disadvantages:

- Bootstrap shell would be a large design dependency.
- Less esports personality than ArenaX needs.

Compatibility:

- Pattern reference only for now.

Recommended components:

- Admin navigation architecture, form grouping, icon discipline, and documentation style.

### TanStack Table

Source: https://github.com/TanStack/table

Advantages:

- Best-in-class headless data-grid logic for sorting, filtering, grouping, selection, pagination, and server-side data.
- Framework adapters support a future React/Vue/Svelte migration.

Disadvantages:

- Not useful without a framework/build pipeline.
- Current app can implement lightweight filters without pulling it in.

Compatibility:

- Strong future candidate once ArenaX has an API-backed admin app.

Recommended components:

- Use its data-table model as inspiration now; adopt later for payment, wallet, audit, and tournament admin tables.

### shadcn/ui

Source: https://github.com/shadcn-ui/ui

Advantages:

- Accessible, customizable component source patterns for dialogs, forms, tabs, cards, tables, and command surfaces.
- MIT license and active ecosystem.

Disadvantages:

- Tailwind/Radix/React-oriented; not directly compatible with the current vanilla stack.

Compatibility:

- Excellent migration target if ArenaX becomes a React/Next app.

Recommended components:

- Component anatomy and accessibility patterns: tabs, dialogs, form fields, badges, toasts, and data cards.

### Tremor

Source: https://github.com/tremorlabs/tremor

Advantages:

- Dashboard and chart components optimized for analytics views.
- Good patterns for KPI cards and revenue analytics.

Disadvantages:

- React/Tailwind dependency.
- Apache-2.0 license is compatible, but source adaptation needs attribution discipline.

Compatibility:

- Future analytics module candidate after framework migration.

Recommended components:

- Revenue cards, payment status charts, and operational analytics layouts.

## Integration Decisions

Integrated now:

- Shared HTML escaping and safe data URL validation for high-risk render paths.
- Safer public/admin tournament rendering for dynamic names, room credentials, rules, emergency messages, result screenshots, participant names, and fraud notes.
- Safer payment rendering for UTRs, notes, filenames, user details, QR images, and screenshot previews.
- Payment server hardening: environment-based path, request size limit, local-only CORS, filename validation, image magic-byte checks, and duplicate-safe destination paths.

Not integrated now:

- No Bootstrap/React/Tailwind admin kit was imported. Direct dependency migration would be too disruptive for the current static codebase.
- No bracket manager/viewer was imported yet. ArenaX's current Free Fire flow is lobby-based and admin-controlled; bracket engines are better after persistence and match-stage modeling exist.
- No billing platform such as Kill Bill was adopted. ArenaX uses manual UPI verification and wallet controls, not automated subscription billing.

## Recommended Migration Plan

Phase 1: Harden current prototype.

- Finish escaping every remaining dynamic `innerHTML` path.
- Replace client-only admin auth with a backend session model.
- Persist tournaments, payments, wallets, audits, and admins in a database.
- Add tests around wallet invariants: no balance mutation without audit and approval.

Phase 2: Build production admin API.

- Node/Express or Fastify backend.
- SQLite/PostgreSQL storage.
- Server-side validation for payment screenshots and tournament actions.
- Role-based authorization for super admin, payment admin, tournament admin, and support.

Phase 3: Upgrade frontend shell.

- Move to React/Vite or Next.js only after APIs exist.
- Use shadcn/ui for accessible primitives, TanStack Table for admin queues, and Tremor-inspired analytics cards.
- Preserve ArenaX visual identity instead of importing a generic admin theme wholesale.

Phase 4: Add bracket/tournament engine.

- Add `brackets-manager.js` for elimination and round-robin stages.
- Add `brackets-viewer.js` or a custom viewer for bracket display.
- Keep Free Fire room/lobby workflows as first-class flows.

# APK_SYSTEM_STATUS.md

Focus: APK release workflow (/api/apk/latest, /admin/apk/upload)

## 1) Frontend behavior (why download button is hidden)
- File: `frontend/public/apk/index.html`
- Download button element: `#downloadBtn` is initially `display: none;`.
- The script calls:
  - `GET /api/apk/latest`
- Button is shown only when response contains an `apk_url`.

### Exact reason download button is hidden
- If `/api/apk/latest` returns **HTTP 404** with message **"No APK releases available."**, then `apk` is not returned as an object containing `apk_url`.
- The frontend catches that condition and calls `showNoRelease()` which:
  - sets `#noApkText` visible
  - keeps `#downloadBtn` hidden

## 2) Backend route status
### GET /api/apk/latest
- Implemented in: `backend/src/modules/apk/apk.routes.js`
- Handler: `backend/src/modules/apk/apk.controller.js#getLatestApk`
- SQL:
  - `SELECT ... FROM apk_versions WHERE is_latest = true ORDER BY created_at DESC LIMIT 1`
- If no row matches `is_latest = true`, backend returns:
  - **404** + **"No APK releases available."**

### Admin upload flow
#### Endpoint
- Defined in: `backend/src/modules/admin/admin.routes.js`
- Route:
  - `POST /admin/apk/upload`
- Middleware chain:
  - `authenticateAdmin`
  - `requireSuperAdmin`
  - `upload.single('apk')`
  - `uploadApkRules`
  - `adminController.uploadApk`

#### Controller upload logic
- Implemented in: `backend/src/modules/admin/admin.controller.js#uploadApk`
- Steps:
  1. Validates `req.file` exists
  2. Uploads file to Supabase Storage via `storage.uploadFile(...)`
     - storage path: `releases/${apkId}.apk`
  3. Deactivates previous latest flags:
     - `UPDATE apk_versions SET is_latest = false`
  4. Inserts record into `apk_versions`:
     - `version_name`, `version_code`, `apk_url`, `changelog`, `is_latest=true`, `uploaded_by`

## 3) Database status (apk_versions rows)
### What we can verify from repo
- Table schema exists: `backend/migrations/015_create_apk_versions.sql`
  - `apk_url TEXT NOT NULL`
  - `is_latest BOOLEAN DEFAULT false`

### Whether table is empty
- **Cannot directly verify row count in this environment** (no DB connection details / credentials available to run a live `SELECT COUNT(*)` here).
- However, based on the frontend behavior described:
  - The backend is returning the 404 message **"No APK releases available."**
  - That means **either**:
    - `apk_versions` has **zero rows**, OR
    - rows exist but **none have `is_latest = true`**.

## 4) Supabase storage status (APK bucket)
### What the backend expects
- Bucket is taken from env:
  - `env.SUPABASE_STORAGE_BUCKET`
- Upload URL generated by backend:
  - returns `${SUPABASE_URL}/storage/v1/object/public/${bucket}/${path}`

### Admin must ensure
- The bucket must exist in Supabase Storage.
- The uploaded path must be publicly accessible (because the backend returns a public URL).
  - If `apk_url` is generated but bucket is not public, clients may fail to download.

## 6) How admin upload generates valid apk_url
- `storage.uploadFile()` uploads to `releases/<uuid>.apk`.
- It returns a **public URL** under `/storage/v1/object/public/...`.
- That returned URL is stored into `apk_url`.

## 7) Summary status
- Route `/api/apk/latest`: **exists and works as designed**, but currently returns **404** because no latest record is available.
- Database: `apk_versions` table exists; row emptiness or `is_latest=false` prevents download.
- Storage: upload path and url generation logic exist; requires correct Supabase bucket + public access.
- Frontend download button: **hidden exactly because** `/api/apk/latest` does not return a record containing `apk_url` (404 No APK releases available).
